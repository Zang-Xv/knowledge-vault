# 缩写规则

问题起因是我总是碰见一些莫名其妙的英文缩写, 但是我半天看不懂也想不出这是个什么?
它缩写的规则似乎不仅仅只是依靠音节, 有时候会莫名出现一些辅音字母
尤其是各种函数命名和命令命名, 所以我想要了解规则以便快速阅读之类的

## 单词缩写方法

### 取词首几位字母（最常见）

只要前 3–5 个字母已经“唯一”了, 就可以停, 就能一眼认出来
或者只保留“关键音节”的前几个字母, 也很有辨识度

| 原词          | 缩写           |
| ------------- | -------------- |
| information   | **info**       |
| application   | **app**        |
| configuration | **config**     |
| administrator | **admin**      |
| documentation | **doc / docs** |
| picture       | **pic**        |
| video         | **vid**        |
| microphone    | **mic**        |
| application   | **app**        |

### 去掉元音a e i o u

多见于技术 / 工程 / 老系统, 一般全部是小写字母的
原因如下:

- 早期打字慢
- 变量名、命令行要短
- 辅音比元音更有辨识度

| 原词     | 缩写    |
| -------- | ------- |
| message  | **msg** |
| password | **pwd** |
| server   | **srv** |

区分度上:

- connection(conn)
- confirmation(cfm/confirm)
- console
- configmap（k8s）

使用conf开头的话难以区分, 太容易撞词了

语义框架上:
元音 让人读、让语言流畅
辅音 承载结构与区分度

音节构造上:
音节的构造一般是辅音开头的, 元音提供音节的主要发音元素
`（起始 Onset） + （韵核 Nucleus） + （韵尾 Coda）`
其中 起始 和 韵尾 是可选的
辅音起始的音节更具有边界感, 元音容易连读, 也就更模糊

统计学上:
出现越多的字母越不具备信息
用出现少的字母反而更有代表性

综上, 我找出一个大致规则:
根据信息逐渐去除字母直到合适长度
首先, 为了更加有辨识度, 会先把元音去除
接下来还会进一步把常见的辅音或者常见韵尾`m n t r s y`去掉, 进一步压缩至只有2-4个字母
我们列出剩下的字母来确认一下, 发现确实是比较常见的命令字母
`b c d f g h j k l p q v w x z`

举例:

1. 来看 configuration 的辅音骨架：
`c  n  f  g  r  t  n`
进一步去掉常见辅音和韵尾
`c f g = cfg`
再填充完整这个词也可以作为取首几位字母的方法
`config`

1. 来看 control 的辅音骨架：
`c  n  t  r  l`
进一步去掉常见辅音和韵尾`n`, 这里不去掉`r`是因为它是`tr`这一起始
`c t r l = ctrl`

### 语言截断

上面的取信息密度高的方法大多用于键盘输入之类的, 还有一部分是要方便交流念出来的
所以会保留发音结构, 进行语言截断

例如: `cat = concatenate`

保留了元音和发音结构, 可以用于日常讨论

### 拉丁 / 历史遗留缩写（规则最怪）

这一类只能“认识”，不必强行推理
看到 `.` 的，大概率是这类

| 缩写 | 原词           | 说明      |
| ---- | -------------- | -----     |
| etc. | et cetera      | 拉丁语    |
| vs.  | versus         | 法律/学术 |
| e.g. | exempli gratia | 举例      |
| i.e. | id est         | 换句话说  |

## 词组缩写方法

### 取每个“实词”的首字母

`the of to`等虚词通常被忽略, 但也还比较好认

| 词组                              | 缩写     |
| --------------------------------- | -------- |
| Hyper Text Transfer Protocol      | **HTTP** |
| Application Programming Interface | **API**  |
| Central Processing Unit           | **CPU**  |
| Object Oriented Programming       | **OOP**  |

### 只取“语义核心词”

偶尔会觉得: 怎么好像少了什么词？

| 原词组                          | 实际缩写 | 被忽略的 |
| ------------------------------- | -------- | -------- |
| Representational State Transfer | **REST** | 无       |
| Quality Assurance Engineer      | **QA**   | Engineer |
| User Experience Design          | **UX**   | Design   |

在语境里：
UX 默认就是 design
QA 默认就是 testing / role

### 首字母组合

聊天缩写之类的多见, 类似于中文中的拼音首字母缩写了

| 缩写       | 含义                   |
| ---------- | ---------------------- |
| TL;DR      | too long; didn’t read  |
| FYI        | for your information   |
| IMO / IMHO | in my (humble) opinion |
| ASAP       | as soon as possible    |

### 修饰 + 核心词汇

第一个词只取一个首字母, 第二个词取 2-3 个字母组成一个长相奇特的单词
大多数是以 修饰词 + 核心 的构造模式

| 全称           | 命令            | 拆法    |
| -------------- | --------------- | ------- |
| secure shell   | **ssh**         | s + sh  |
| stream editor  | **sed**         | s + ed  |
| super user do  | **sudo**        | su + do |
| network mapper | **nmap**        | n + map |

其中的`shell -> sh`与`editor -> ed`已经作为命令缩写深入人心(虽然我在此之前并不认得)
所以只需要加上一些分类修饰就很好认出, 例如 csh / zsh / bash 都在这个谱系上
